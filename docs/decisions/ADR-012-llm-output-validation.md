# ADR-012: LLM Output Validation (Phase 2, deferred)

## Status

Proposed — Deferred

이 ADR은 결정을 지금 확정하지 않는다. 무엇을 결정해야 하는지, 왜 지금이 아닌지, 어떤 신호가
오면 확정해야 하는지를 기록한다. 풀구현 트리거가 충족되면 본 문서를 갱신해 Status를
`Proposed`(이후 `Accepted`)로 올린다.

## Context

LLM 출력은 불확정적이다. 구조화 출력(`complete_json`, ADR-007의 기존 transport 기능)을
쓰더라도 스키마에 맞지 않거나, 맞더라도 내용이 부적절한 결과가 나올 수 있다. 스펙(§7)은
검증을 통과한 결과만 저장하고, 실패하면 safe template로 대체할 것을 요구한다.

1. **스키마 검증.** 결과가 기대한 구조(필드·타입·열거값)를 만족하는지 확인한다.
2. **저장 게이트.** 검증을 통과한 결과만 영속화한다. 깨진 결과가 DB에 들어가 하류 기능을
   오염시키지 않게 한다.
3. **safe template.** 검증 실패 시 사용자에게 보여줄 안전한 기본 응답.

**현재 상태부터 정확히 짚는다.** "스키마 검증 + 통과분만 저장"의 기본형은 이미 작동 중이다.
뉴스 AI 요약(`app/domains/news/service.py`)은 `complete_json(messages, NewsSummaryResult)`으로
출력을 받아 Pydantic 스키마로 검증하고, 통과한 결과만 `news_items`에 저장하며, 실패하면
업데이트하지 않고 예외를 전파한다(설계 014). 즉 "검증 대상 스키마가 없다"는 사실이 아니다.

본 ADR이 풀로 설계하지 않고 미루는 것은 그 위에 얹는 세 가지다.

- **게이트웨이 레벨 일반화.** 현재 검증은 호출부(서비스)마다 개별로 이뤄진다. 이를
  게이트웨이라는 단일 지점으로 끌어올려 모든 클라우드·로컬 작업에 일관 적용하는 일.
- **내용 수준 검증.** `complete_json`은 형태(필드·타입·열거값)는 보장하지만, 값 범위·금칙어·
  일관성 같은 내용 적정성은 보지 않는다.
- **safe template 폴백.** 현재 실패 처리는 예외 전파다(워커 잡 맥락에선 충분하다). 사용자
  대면 기능은 예외 대신 안전한 기본 응답을 보여줘야 한다.

이 셋의 구체적 형태는 소비 기능(브리핑)이 어떤 구조의 결과를 요구하고 실패 시 무엇을 보여줄지가
정해져야 잡힌다. 기능 없이 범용 검증 프레임워크를 먼저 만들면 투기적 추상화가 된다(Epic #141의
Phase 분리 근거).

## Decision

지금은 확정하지 않는다. 대신 풀구현 시점에 지켜야 할 제약과 트리거만 고정한다.

1. **풀구현 트리거.** 형태 검증을 넘어 (a) 내용 수준 검증이나 (b) 예외 전파 대신 safe template
   폴백이 필요한 소비처가 등장할 때 — 사실상 포트폴리오·대시보드 브리핑처럼 사용자 대면으로
   결과를 보여주는 기능 설계 착수 시점이다. 출력을 영속화하는 작업(뉴스 요약)은 이미 존재하지만,
   호출부 레벨의 형태 검증 + 예외 전파로 충분히 다뤄지므로 그 자체로는 트리거가 아니다.
2. **확정 시 지켜야 할 제약.**
   - 검증은 게이트웨이 레벨에서 수행하여 모든 클라우드·로컬 결과에 일관 적용한다. 호출부가
     자체 검증을 중복 구현하지 않는다(ADR-007의 계층 분리와 일관).
   - 검증 통과분만 저장한다. 실패한 결과는 저장하지 않고 safe template로 대체한다.
   - safe template로의 대체는 ADR-010의 품질 폴백과 같은 경로로 처리한다. "신뢰할 수 없는
     결과"의 종착지를 두 ADR이 다르게 두지 않는다.
   - 검증 실패는 관측 가능해야 한다(로깅·메트릭). 모델 품질 저하나 프롬프트 회귀를 조용히
     숨기지 않는다.
3. **확정해야 할 미결 질문.**
   - 검증 강도 — `complete_json`의 스키마 적합성으로 충분한지, 내용 수준(값 범위·금칙어·
     일관성) 검증까지 둘지.
   - safe template를 작업별로 둘지 공통 형태로 둘지.
   - 검증 실패율의 관측 지표와 경보 임계치.

## Alternatives

지금 평가하지 않고, 확정 시 비교할 후보로만 기록한다.

- **`complete_json` 스키마 검증에만 의존(현재 상태 유지).** 형태는 보장하나 내용 부적절은 못
  거르고, 실패는 예외 전파로만 처리한다. 뉴스 요약 같은 워커 잡에는 이것으로 충분하다.
- **검증 실패 시 저장하되 플래그만 표시.** 기각 방향이 유력하다. 깨진 결과가 DB에 남아 하류를
  오염시킨다. 스펙의 "통과분만 저장" 요구와 어긋난다.
- **호출부별 검증.** 기각 방향이 유력하다. ADR-007/008과 같은 이유로 정책을 흩뜨린다.

## Consequences

- 보류의 효과: 출시 경로는 호출부 레벨의 `complete_json` 형태 검증과 예외 전파로 동작한다.
  게이트웨이 레벨의 내용 검증·safe template은 두지 않는다. 뉴스 요약 같은 워커 잡에는 문제가
  되지 않는다.
- 리스크: 사용자 대면 브리핑이 예외 전파만으로는 부족한 실패 경험을 만들거나, 형태는 맞지만
  내용이 부적절한 결과를 그대로 보여줄 수 있다. 트리거를 명시해 도입 시점을 관리한다.
- 코드·DB 변경 없음. 본 문서는 결정 보류 기록이다.

## Follow-up

- 브리핑 기능 설계(Phase 2) — 검증 대상 스키마와 저장 요구를 확정하는 선행 의존성.
- ADR-010 — safe template 대체 경로(품질 폴백)를 공유한다.
