`#48`은 같은 작업을 두 트랙으로 각각 끝까지(구현+로컬리뷰) 수행한 통제된 A/B다.
산출 규모도 동등: Track A 169파일/+16,842, Track B 170파일/+17,126.

## 핵심 발견 1 — VFF는 Sonnet으로 돌지 않았다

세션 로그 전수 조사 결과 2026-06-23 이후 **비-opus 모델 사용이 0건, 서브에이전트(isSidechain) 호출도 0건**이다. VFF는 별도 Sonnet 서브에이전트가 아니라 **메인 Opus 세션에 itsvff 세션모드를 적용**한 형태로 동작했다. itsvff 스킬은 동작 규율만 입히고 모델을 교체하지 않으므로, "Sonnet 단가로 인지작업을 싸게 처리"하는 이득은 **실현되지 않았다**. 두 트랙 모두 전 구간 Opus 단가다.

## 핵심 발견 2 — 토큰·비용 실측 (#48 동일 작업)

Opus 4.8 단가(USD/1M: in 15 / out 75 / cacheW 18.75 / cacheR 1.5) 기준.

| 지표 | A: 순수 Opus | B: Opus+VFF | 차이 |
| --- | --- | --- | --- |
| 턴 수 | 64 | 88 | +37% |
| output 토큰 | 75,075 | 115,046 | +53% |
| 총 토큰(캐시 포함) | 7.54M | 9.43M | +25% |
| 비용 | $18.62 | $29.12 | **+56%** |

VFF 트랙이 같은 산출물을 내는 데 토큰·비용 모두 더 들었다. 모델 단가가 같은 상태에서 VFF의 구조화·검토 패스와 정정 루프가 순증으로 작용했다.

(보조: #47 VFF 트랙 = 95턴 / output 129,680 / 19.64M / $53.85. #48 대비 작업 규모가 커 단독 비교 부적합, 참고치.)

## 핵심 발견 3 — 산출물 품질 (동일 이슈, path-prefix 처리)

두 트랙 모두 `/api/v1` prefix 불일치라는 같은 결함을 만났다.

- **Track A(pr-58)**: 더블 프리픽스(`…/api/v1/api/v1/…`) 위험을 정확히 진단하고 prefix 제거로 **첫 패스에 수정**, 재검증 통과.
- **Track B(pr-59)**: VFF 리뷰가 "prefix를 추가하라"고 **방향을 반대로 진단**(적용 시 더블 프리픽스 404). Opus 최종검증에서 정정 — 리뷰 산출물에 correctness 오류가 섞였고 Opus 정정 패스가 필수였다.

VFF의 갭 처리·어댑터·테스트·스코프 평가 깊이(G-1~G-6)는 정밀했으나, 리뷰의 **신뢰성이 보장되지 않아** Opus 최종검증 없이는 채택 불가다.

## 결론

이 워크로드에서 Track B는 Track A 대비 **비용 +56%, 토큰 +25%이면서 동등 산출물**이고, 기대했던 Sonnet 단가 이득은 구조상 발생하지 않았으며, 리뷰 신뢰성은 오히려 정정 패스를 요구했다. 따라서 현재 하니스 구성(VFF=Opus 세션모드)에서는 **Track A(순수 Opus+Codex)가 비용·신뢰성 모두 우위**다.

VFF가 의미를 가지려면 (1) 실제 Sonnet/저단가 모델 세션에서 구동되어 단가 차이를 만들거나, (2) Opus 최종검증을 생략해도 될 만큼 리뷰 신뢰성이 확보되어야 한다. 둘 다 현재 충족되지 않았다. → 후속: VFF를 실제 Sonnet 세션에서 구동하는 환경으로 BE에서 재측정 예정.
