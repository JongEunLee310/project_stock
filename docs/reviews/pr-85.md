## Review Summary

PR #85 — Issue #63 `[BE] Health Check API 개선` 구현

핸드오프/설계(043) 범위를 정확히 충족한다. liveness와 readiness를 분리했다. `GET /health`는 의존성을 점검하지 않고 `{"status":"ok"}`를 빠르게 반환해 기존 모니터링 호환을 유지한다. `GET /health/readiness`는 DB 세션으로 `SELECT 1`을 실행해 연결을 점검하고, 실패 시 503과 함께 `status="error"`/`checks.db="error"`를 반환한다. 응답에는 provider mode(market/news/disclosure/portfolio)와 application version이 포함된다. version 소스는 설치 패키지 메타데이터(`importlib.metadata.version`) 우선, 미설치 시 `pyproject.toml` fallback의 단일 소스다.

`app/main.py`의 중복 `/health` 인라인 정의를 제거하고 health 라우터로 일원화했다(설계 Current State의 중복 정의 해소). 보호 파일 변경 없음. 신규 도메인/스키마 변경은 health 응답 모델(`ReadinessResponse` 등)에 한정되며 공통 envelope·기존 API 스키마는 무변경. 로컬 검증 전부 통과(208 passed), CI(`Verify Project Stock`) pass. 블로킹 이슈 없음.

## Suggestions

### 1. health 라우터가 `/health`와 `/api/v1/health` 양쪽에 등록 — readiness도 양쪽 노출 (관찰)

**위치:** `app/main.py:34`, `app/api/v1/router.py:19`

동일한 health 라우터가 루트(`application.include_router(health.router, prefix="/health")`)와 api_router(`/api/v1/health`) 양쪽에 등록된다. 결과적으로 `/health/readiness`와 `/api/v1/health/readiness`가 모두 존재한다. 설계는 루트 경로(`/health`, `/health/readiness`)를 의도했고 `/api/v1/health`는 기존부터 존재하던 경로다. 동작상 해롭지 않으며 테스트도 `/api/v1/health` 200을 검증한다.

**권장 수정 방향:** 유지. 외부 노출 경로를 단일화하고 싶다면 #65 통합 문서에서 "운영 모니터링은 루트 `/health*` 사용" 컨벤션을 한 줄 명시하면 충분하다.

### 2. readiness는 DB만 점검 — Redis(worker 의존성)는 미포함 (의도된 범위)

**위치:** `app/api/v1/endpoints/health.py:53-72`

설계 Decisions에서 Redis 포함 여부 및 hard fail/degraded 처리를 구현 시 결정으로 남겼고, 본 구현은 DB만 hard fail(503) 대상으로 한다. 현재 범위에서 적절하다. worker 의존성 가시성이 필요해지면 후속에서 `checks.redis`를 degraded 신호로 추가하는 방향을 권장.

**권장 수정 방향:** 유지. 후속 과제로 기록만.

### 3. version fallback의 pyproject 파일 경로 의존 (Minor)

**위치:** `app/api/v1/endpoints/health.py:14, 81-89`

`PROJECT_ROOT = Path(__file__).resolve().parents[4]`로 pyproject 위치를 역산한다. 패키지가 정상 설치된 환경에서는 메타데이터 경로를 타므로 fallback은 개발/미설치 환경 한정이다. 디렉터리 깊이 변경 시 `parents[4]`가 깨질 수 있으나 현재 구조에서는 정확하다.

**권장 수정 방향:** 유지. 향후 패키지명/구조 변경 시 메타데이터 경로가 우선이라 영향은 제한적.
