# PR #162 — 관심종목 요약 추이 시계열 endpoint (feat/watchlist-summary-trends)

리뷰어: Claude Code (Opus, 오케스트레이터). 구현: Codex. 브랜치: feat/watchlist-summary-trends → main. 관련: BE #161, 설계 063, 미러 선례 062, unblock FE #96.

## Review Summary

설계 063대로 `GET /api/v1/watchlists/{watchlist_id}/summary/trends`를 신설했습니다. 대시보드
추이(062)의 날짜 윈도우·0채움·응답 와이어 형태를 미러하되, watchlist ownership 가드와 두
파생 계열(`watchlist_total`·`risk_increasing`)을 현재 watchlist 코호트 기준 as-of로 계산합니다.

`WatchlistSummaryTrendService.get_trends`는 `_get_owned_watchlist`로 ownership을 확인하고(미존재
404·비소유 403), 현재 항목의 `(asset_id, created_at)`로 `watchlist_total`(각 날짜 D에
`created_at <= as_of` 항목 수)을, `SignalRepository.count_assets_with_active_signal_as_of`로
`risk_increasing`(각 날짜 D에 활성 RISK_ALERT 보유 고유 자산 수)을 계산해 0채움 후 반환합니다.

`_as_of_at`은 오늘은 `utc_now()`, 과거일은 날짜 종료 경계(다음날 00:00 UTC)를 기준으로 삼아,
마지막 날 값이 기존 summary의 현재 활성 판정과 일치하도록 보장합니다. 이는 설계 §3의 "마지막
날 일관성" 요구를 정확히 충족합니다.

설계의 결정이 모두 반영됐습니다: (1) watchlist 도메인 독립 스키마(대시보드 스키마 미import,
와이어 형태 동일), (2) 현재 코호트 as-of 파생, (3) 마지막 날 == 현재 total_count·
risk_increasing_count, (4) 0채움·빈 watchlist 전 구간 0, (5) ownership·days 경계.

## Blocking

None.

## Suggestions

**S1 · as-of 파라미터 타입(비차단)**

`count_assets_with_active_signal_as_of(..., as_of: Any)`가 `Any` 타입입니다. 실제로는
`datetime`이므로 향후 `datetime`으로 좁히면 타입 명확성이 개선됩니다. mypy는 통과하며 동작에
영향이 없어 비차단입니다.

**S2 · protected 메서드 재사용(비차단, 설계 근거)**

trend 서비스가 `WatchlistService._get_owned_watchlist`(protected)를 직접 호출합니다. ownership
가드 재사용이라는 설계 결정에 따른 것으로, 중복 구현을 피하는 실용적 선택입니다. 공용 가드
추출은 후속 여지가 있으나 현 단계 비차단입니다.
