## Review Summary

PR #86 — Issue #64 `[BE] 스케줄러/잡 실행 구조 초안 추가` 구현

핸드오프/설계(044) + ADR-003(Accepted, rq-scheduler) 범위를 정확히 충족한다. scheduler 구조를 초안 수준으로 도입했다.

- **Job interface** (`app/scheduler/interface.py`): `SchedulerJob` Protocol(`name`/`run()`)와 `FunctionSchedulerJob`(frozen dataclass)로 기존 함수형 job을 감싸는 최소 인터페이스. 설계의 "name + run() 수준" 요구와 일치.
- **Registry** (`app/scheduler/registry.py`): `ScheduleDefinition`(job·cron·enabled) + `SchedulerRegistry`(name→정의 맵, get/list). `default_scheduler_registry`에 `mock_collection`(cron `*/15 * * * *`) 선언적 등록. 설계의 schedule registry 단일 지점과 일치.
- **mock job** (`app/scheduler/jobs.py`): 외부 연동 없이 로그만 남기는 흐름 확인용.
- **ManualSchedulerRunner** (`app/scheduler/runner.py`): 수동 1회 실행. `JobRunService.start(name, metadata)` → `run()` → `succeed`/`fail` 흐름으로 `job_runs`에 기록하고 metadata(trigger=manual/cron/scheduler=mock)를 첨부. 미등록은 `SchedulerJobNotFoundError`, disabled는 `SchedulerJobDisabledError`로 구분.
- **수동 실행 endpoint** (`app/api/v1/endpoints/worker.py`): `POST /scheduler/jobs/{job_name}/run`(worker 라우터 prefix 기준 `/api/v1/worker/scheduler/jobs/{job_name}/run`). 404(미등록)/409(disabled) 매핑.

`JobRunService` 시그니처(`start`/`succeed`/`fail`)와 runner 호출이 정확히 일치한다. 기존 worker/job_runs 인프라를 대체하지 않고 그 위에 얹는 구조다. 실제 주기 트리거는 mock으로 유지(rq-scheduler 전환은 후속). 보호 파일 변경 없음. 로컬 검증 전부 통과(213 passed), CI pass. 블로킹 이슈 없음.

## Suggestions

### 1. 수동 실행 경로가 `worker` 라우터에 포함됨 (관찰)

**위치:** `app/api/v1/endpoints/worker.py:66-98`

scheduler 수동 실행 endpoint가 worker 라우터에 추가되어 실제 경로가 `/api/v1/worker/scheduler/jobs/{job_name}/run`이 된다. 설계는 "기존 `POST /jobs/*` 흐름 재사용"을 의도했으므로 worker 모듈 배치는 합리적이나, scheduler 도메인이 커지면 별도 라우터(`scheduler.py`)로 분리하는 편이 경로 일관성에 유리하다. 현재 초안 범위에서는 적절.

**권장 수정 방향:** 유지. scheduler 기능 확장 시 #65 통합 문서에서 경로 컨벤션 정리.

### 2. ManualSchedulerRunner가 `default_scheduler_registry`에 직접 결합 (의도된 범위)

**위치:** `app/api/v1/endpoints/worker.py:76-79`

endpoint가 `default_scheduler_registry`를 직접 참조한다. 테스트에서는 runner를 직접 구성해 임의 registry를 주입하므로 단위 테스트 격리는 확보됐다. 초안 단계에서 DI 추상화를 더 두지 않은 것은 "현 단계에 불필요한 추상화 지양" 원칙에 부합.

**권장 수정 방향:** 유지. 실제 주기 등록 전환 시 registry 주입 지점 재검토.

### 3. mock job의 결과 가시성은 로그·job_runs에 한정 (의도된 범위)

**위치:** `app/scheduler/jobs.py:6-7`

mock job은 로그만 남기고 별도 산출물이 없다. job_runs 기록과 endpoint 응답(job_run_id/status)으로 흐름은 충분히 검증된다. 실제 수집 로직은 후속 범위이므로 적절.

**권장 수정 방향:** 유지. 후속 과제로 기록만.
