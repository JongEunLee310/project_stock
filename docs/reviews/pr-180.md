## Review Summary

LLM 데이터 파이프라인 2단계로, 기존 원본 저장 테이블 두 곳에 가공 진행 상태 컬럼을 더한 PR입니다.
`raw_prices`·`raw_news_events` 모델에 `processing_status`(String(20), not null, index)를 추가하고,
기본값을 `server_default="fetched"`로 두어 신규 insert와 기존 행 backfill을 모두 처리합니다. 값
도메인은 1단계에서 정의한 `app/domains/ingestion/schema.py`의 `ProcessingStatus`를 그대로 import해
재사용하며, 병렬 enum을 새로 만들지 않았습니다(설계 §3).

마이그레이션 `c3d4e5f60058`은 `down_revision`이 1단계 head `c3d4e5f60057`이라 단일 head를
유지하고, upgrade에서 두 컬럼과 두 인덱스(`ix_raw_prices_processing_status`,
`ix_raw_news_events_processing_status`)를 추가하며, downgrade가 이를 대칭으로 되돌립니다. 수집
서비스(`save_raw`)와 repository에는 상태 값을 주입하거나 전이시키는 코드를 넣지 않아, 상태 전이는
3·4단계로 남긴 설계 범위를 정확히 지켰습니다. 보호 파일(`app/domains/ingestion/*`,
`app/adapters/llm/*`, 기타 도메인)은 건드리지 않았고 `ProcessingStatus`는 재사용만 했습니다.

설계 Decision E(`skipped_duplicate`는 dedup 시 행이 저장되지 않아 영속 행 상태가 아님)도 반영돼,
CHECK 제약 없이 문자열로만 저장합니다. 테스트는 서비스 경유 insert의 기본값 `fetched`, `normalized`/
`failed` 명시 저장·조회, 마이그레이션 up/down 회귀(컬럼·인덱스 생성/삭제, 기존 행 backfill)를
모두 덮습니다.

## Suggestions

- **S1 · 상태 값 무결성 강제 시점**: 현재 컬럼은 자유 문자열이고 값 정합은 애플리케이션 계층
  (`ProcessingStatus` 재사용)에만 의존합니다. Decision E대로 CHECK 제약을 두지 않은 선택은
  타당합니다. 다만 3단계에서 상태별 조회·재처리가 붙으면, 잘못된 문자열이 유입되지 않도록 전이
  경로를 enum을 거치도록 강제하는 것을 그때 함께 확인하면 좋겠습니다.
- **S2 · 컬럼 위치**: `RawPrice`는 `processing_status`를 `fetched_at` 앞에, `RawNewsEvent`는
  `payload` 앞에 두어 테이블별 배치가 다릅니다. 마이그레이션으로 추가되는 컬럼이라 물리 순서에는
  영향이 없고 기능상 문제는 없습니다. 통합 원본 테이블로 합치는 시점(1단계 Decision A 후속)에
  컬럼 순서를 한 번 정렬하면 됩니다.

## Questions

- **Q1 · 상태별 조회 인덱스 활용**: 이번에 추가한 단일 컬럼 인덱스는 3단계에서 "가공 대기
  (`fetched`) 행" 질의를 대비한 것입니다. 실제 소비 쿼리가 `processing_status` 단독인지, 아니면
  `(processing_status, fetched_at)` 같은 복합 조건인지에 따라 3단계에서 복합 인덱스로 승격이
  필요할 수 있습니다. 2단계 범위로는 단일 인덱스가 맞다고 봅니다.
