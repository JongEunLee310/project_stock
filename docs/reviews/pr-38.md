## Suggestions

### 1. `create_if_absent` 멱등성 의존 방식 (Minor)

**위치:** `app/domains/alerts/repository.py:13-26`

DB 유니크 제약 위반(`IntegrityError`)을 잡아 중복을 멱등 처리하는 방식은 견고하다. 다만 동일 세션에서 commit 실패 후 rollback하므로, 호출부가 같은 트랜잭션에서 다른 작업을 묶고 있다면 함께 롤백된다. 현재는 Alert 생성이 독립 경로라 문제없으나, 향후 #19 통합 흐름에서 한 트랜잭션에 여러 Alert를 묶을 경우 부분 실패 격리를 재검토할 필요가 있다.

### 2. `dedup_key`의 `news_item_id` None 케이스 (Minor)

**위치:** `app/domains/alerts/service.py:20`

`news_item_id`가 없는 Signal(예: 일부 규칙 미사용 경로)에서 dedup_key가 `"RISK_ALERT:None"`이 된다. 현재 두 규칙 모두 `news_item_id`를 채우므로 실제 위험은 없으나, news_item 없는 Signal이 추가되면 dedup 의미가 약해질 수 있다. 현 단계 유지 무방.
