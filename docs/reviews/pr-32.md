## Suggestions

### 1. `JobRunService.start()` — 불필요한 이중 커밋 (Minor)

**위치:** `app/domains/jobs/service.py:14-20`

`start()` 내부에서 `repo.create()` (status="pending", 커밋) → `repo.update_status()` (status="running", 커밋) 순서로 DB 왕복이 두 번 발생한다. 설계 문서의 의도는 `start()` 반환 시 `status=running`인 단일 상태이므로 중간의 "pending" 상태는 불필요하다.

**권장 수정 방향:** `repo.create()`가 `status` 파라미터를 받도록 수정하거나, `start()` 내부에서 직접 `status="running"`, `started_at=now()`로 객체를 생성.

현 구현도 기능상 정확하고 테스트 통과. 필수 수정 아님 — 다음 리팩터링 기회에 정리.

### 2. `JobRunCreate` 스키마 미사용 (Minor)

**위치:** `app/domains/jobs/schema.py:7-9`

`JobRunCreate`가 정의되어 있으나 `JobRunRepository.create()`는 raw 파라미터(`job_type, metadata`)를 직접 받는다. 다른 도메인(`RawNewsEventCreate`, `NewsItemCreate`)은 repository가 스키마 객체를 받는 패턴을 쓰고 있어 불일치가 있다.

task-010(Worker enqueue API)에서 사용될 가능성이 있어 현 상태로 유지 가능. task-010 완료 후에도 사용되지 않는다면 제거하거나 repository에 적용.
