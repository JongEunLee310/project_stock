## Suggestions

### 1. alert candidates에 신규 `sort` query param 추가 (Minor)

**위치:** `app/api/v1/endpoints/alert_candidates.py`

알림 후보 목록에 `sort`(허용 `created_at`/`id`, 기본 `-created_at`) query parameter가 새로 노출된다. 기존 호출자가 `sort`를 생략하면 기본값이 종전 정렬(`created_at desc, id desc`)과 동일해 기존 동작은 보존된다. 설계 036이 알림 후보에 정렬 규칙 적용을 명시하므로 범위 내 변경 — 문제 없음. (응답 구조 contract 고정은 #61 범위.)

**권장 수정 방향:** 유지.

### 2. `sort_param`의 default 검증이 반환값을 버림 (Minor)

**위치:** `app/core/pagination.py:72`

`sort_param`은 모듈 로드 시 `parse_sort(default, ...)`를 호출해 잘못된 default를 조기에 422로 드러내는 가드를 둔다. 반환값은 사용하지 않고 검증 목적만 갖는다 — 의도된 fail-fast로 적절. ruff 통과.

**권장 수정 방향:** 유지.

### 3. `parse_sort`의 `RequestValidationError` 페이로드 형태 (Minor)

**위치:** `app/core/pagination.py:53`

허용되지 않은 필드는 `RequestValidationError([{loc, msg}])`로 던진다. pydantic 표준 error dict의 `type`/`input` 키는 없지만 프로젝트 예외 핸들러가 422 `VALIDATION_ERROR` envelope로 매핑함을 테스트(`test_list_alert_candidates_rejects_invalid_sort`)로 확인. 동작에 문제 없음.

**권장 수정 방향:** 유지.
