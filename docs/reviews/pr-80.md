PR #80 — Issue #59 `[BE] DB Migration 구조 정리` 구현

핸드오프/설계(039) 범위를 정확히 충족한다. `alembic/env.py`에 누락돼 있던 `alerts`/`signals`/`reports`/`theses.conflict_model` import를 보완해 전체 14개 도메인 model이 `Base.metadata`에 등록되도록 했고, single head를 유지하며, 마이그레이션 절차를 README에 문서화했다. import 집합·테이블 등록을 강제하는 메타데이터 테스트를 추가했다. 기존 마이그레이션 재작성·스키마 변경 없음. 로컬 검증 전부 통과. 블로킹 이슈 없음.

## Suggestions

### 1. 메타데이터 테스트가 import 일치를 단방향으로만 강제 (Minor)

**위치:** `tests/test_alembic_metadata.py:30`

`test_alembic_env_imports_all_domain_models`는 `도메인 model 집합 ⊆ env.py import 집합`(부분집합)을 단언한다. 도메인 model이 추가되면 env.py 누락을 잡아내는 의도된 가드로 충분하다. 역방향(env.py에만 있는 잉여 import) 검출은 현재 불필요 — 잉여 import는 autogenerate 정확도에 해를 주지 않는다. 현 상태 유지.

**권장 수정 방향:** 유지.

### 2. `glob("**/*model.py")` 패턴의 범용성 (Minor)

**위치:** `tests/test_alembic_metadata.py:16`

`*model.py` 글롭은 `conflict_model.py`처럼 접미사가 `model.py`인 파일도 포착한다 — 의도대로 동작(conflict_model 포함 14개 전부 인식). 향후 `*_model_helper.py` 같은 비모델 파일이 생기면 오탐 가능성이 있으나 현재 네이밍 규약상 문제 없음. 현 상태 유지.

**권장 수정 방향:** 유지.
