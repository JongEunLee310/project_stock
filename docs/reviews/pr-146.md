# PR #146 로컬 리뷰

리뷰어: Claude Code (Opus, BE 트랙). 구현: Codex(`codex exec`, 핸드오프 task-120). 브랜치: feat/llm-router-factory → main.

## Review Summary

이슈 #134 구현입니다. provider 선택을 작업 유형(task type) 기반으로 확장합니다. 작업 유형에 따라 provider를 고르는 `LLMRouter`, `LLM_PROVIDER` 설정, factory `get_llm_client()` 배선을 추가하고, worker job이 인라인으로 만들던 `MockLLMClient(...)`를 factory 경유로 정리합니다. 설계 054·핸드오프 task-120의 확정된 계약과 ADR-007~008의 결정에 일치합니다.

- `app/adapters/llm/router.py`: 불변 값 객체 `TaskRoute`, 모듈 수준 라우팅 표 `LLM_TASK_ROUTES`, `LLMRouter`를 추가합니다. 표는 `LLMTaskType`의 7개 멤버를 모두 포함하며, 출시 시점에는 `launch`가 전부 `cloud`이고 작업별로 달라지는 값은 `future_primary`뿐입니다(ADR-008 §3.3·§4). 라우팅을 `if` 분기가 아니라 표 데이터로 표현했습니다. `resolve`는 표에 없는 task_type을 받으면 조용히 기본값으로 넘기지 않고 `LLMRoutingError`를 던져 fail-closed로 동작합니다.
- `app/core/config.py`: `LLM_PROVIDER: Literal["cloud","local","mock"] = "cloud"`를 추가합니다. 기존 `*_PROVIDER` 설정 패턴과 ADR-008 §3.2를 따릅니다.
- `app/adapters/factory.py`: `get_llm_client(provider=None)`을 추가합니다. `settings.LLM_PROVIDER`를 기준으로 transport를 생성하며, cloud인데 키가 없으면 `RuntimeError`로 명확히 실패합니다(시스템 경계 처리). 기존 `get_*_provider`와 같은 스타일입니다.
- `app/adapters/llm/mock.py`: 캔드 응답을 `DEFAULT_MOCK_RESPONSES` 상수로 옮겼습니다. 값과 구조를 그대로 보존해 dev mock 산출에 회귀가 없습니다.
- `app/adapters/llm/exceptions.py`: `LLMRoutingError(LLMCallError)`를 추가합니다. 기존 예외 계열과 일관됩니다.
- `app/worker/jobs/analysis.py`: 인라인 mock 생성과 그 import를 제거하고 `get_llm_client()`를 경유하도록 바꿉니다.
- `.env.example`: 로컬 dev가 mock을 택하도록 `LLM_PROVIDER=mock`을 추가합니다.

## Suggestions

**S1 · 라우터와 `LLM_PROVIDER`의 결합은 #136으로 이연** — Phase 1에서 `resolve`는 모든 작업에 `cloud`를 반환하고, dev에서 mock으로 전역 전환하는 일은 `LLM_PROVIDER` 설정(factory)이 담당합니다. 이 둘을 묶는 결합 정책(예: dev의 mock 설정이 라우터 결과보다 우선하는지)은 gateway(#136)가 소유하며 본 PR 범위가 아닙니다. 설계 054 §5에 명시돼 있어 의도된 경계입니다. 비차단.

**S2 · `provider` 인자 타입** — `get_llm_client(provider: str | None)`는 자유 문자열을 받고, 모르는 값이 들어오면 `NotImplementedError`로 막습니다. 합리적입니다. 호출부가 생기는 #136에서 라우터가 만든(검증된) provider 문자열을 넘기게 되므로, 현 단계에서 타입을 `Literal`로 좁힐 필요는 없습니다. 비차단.
