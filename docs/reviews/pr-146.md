리뷰어: Claude Code (Opus, BE 트랙). 구현: Codex(`codex exec`, 핸드오프 task-120). 브랜치: feat/llm-router-factory → main.

## Review Summary

이슈 #134 구현. provider 선택을 task 인지형으로 확장합니다. `LLMRouter`(task_type→provider 데이터 주도 매핑), `LLM_PROVIDER` 설정, factory `get_llm_client()` 배선을 추가하고, worker job의 인라인 `MockLLMClient(...)` 생성을 factory 경유로 정리합니다. 설계 054·핸드오프 task-120의 동결된 계약 및 ADR-007~008과 정확히 일치합니다.

- `app/adapters/llm/router.py`: 불변 값 객체 `TaskRoute`, 모듈 수준 라우팅 표 `LLM_TASK_ROUTES`, `LLMRouter`를 추가합니다. 표는 `LLMTaskType`의 7개 멤버를 모두 포함하며, 출시 시점에는 `launch`가 전부 `cloud`이고 작업별로 달라지는 값은 `future_primary`뿐입니다(ADR-008 §3.3·§4). 라우팅을 `if` 분기가 아니라 표 데이터로 표현했습니다. `resolve`는 표에 없는 task_type을 받으면 조용히 기본값으로 넘기지 않고 `LLMRoutingError`를 던져 fail-closed로 동작합니다.
- `app/core/config.py`: `LLM_PROVIDER: Literal["cloud","local","mock"] = "cloud"`. 기존 `*_PROVIDER` 패턴·ADR-008 §3.2와 일치.
- `app/adapters/factory.py`: `get_llm_client(provider=None)` — `settings.LLM_PROVIDER` 기준 transport 생성. cloud 키 부재 시 `RuntimeError`로 명확히 실패(시스템 경계 처리). 기존 `get_*_provider` 스타일 답습.
- `app/adapters/llm/mock.py`: 캔드 응답을 `DEFAULT_MOCK_RESPONSES` 상수로 이전. 값·구조 보존으로 dev mock 산출 회귀 없음.
- `app/adapters/llm/exceptions.py`: `LLMRoutingError(LLMCallError)` — 기존 예외 계열 일관.
- `app/worker/jobs/analysis.py`: 인라인 mock 생성·import 제거 → `get_llm_client()` 경유.
- `.env.example`: `LLM_PROVIDER=mock`(로컬 dev opt-in).

## Suggestions

**S1 · 라우터의 `LLM_PROVIDER` 결합은 #136으로 이연** — Phase 1에서 `resolve`는 전 작업 `cloud`를 돌려주고, dev에서 mock으로 전역 전환하는 것은 `LLM_PROVIDER` 설정(factory)이 담당합니다. 이 둘을 묶는 결합 정책(예: dev에서 mock override가 라우터 결과를 우선하는지)은 gateway(#136)가 소유하며 본 PR 범위가 아닙니다. 설계 054 §5에 명시돼 있어 의도된 경계입니다. 비차단.

**S2 · `provider` 인자 타입** — `get_llm_client(provider: str | None)`는 자유 문자열을 받고 미지 값에 `NotImplementedError`로 fail-closed합니다. 합리적입니다. 호출부가 생기는 #136에서 라우터 결과(검증된 provider 문자열)를 넘기게 되므로 현 단계에서 `Literal` 협소화는 불필요합니다. 비차단.
