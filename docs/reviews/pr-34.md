# PR #34 로컬 리뷰 — LLM Adapter 기본 구조 구현 (Issue #13)

## Review Summary

Issue #13 요구사항을 전부 충족한다. `LLMClient` 추상화 레이어가 올바르게 설계되어 Service가 특정 LLM 제공자에 직접 의존하지 않는다. `MockLLMClient`로 테스트가 가능하며, `OpenAIClient` 호출 실패 시 `LLMCallError`/`LLMTimeoutError`로 변환되어 시스템이 중단되지 않는다. lint, typecheck, 테스트 모두 통과.

## Suggestions

**1. `complete_json` 오류 경로 테스트 누락**

`OpenAIClient.complete_json`의 아래 경로가 테스트되지 않았다 (coverage 미반영 라인: 56-59, 67-68, 71).

- `APITimeoutError` → `LLMTimeoutError` 변환
- `OpenAIError` → `LLMCallError` 변환
- 응답이 유효하지 않은 JSON인 경우
- 응답이 JSON 객체가 아닌 경우 (배열 등)

`complete` 메서드와 동일한 오류 처리 로직이므로 Issue #14 작업 시 `tests/test_news_analysis.py`에서 통합 커버리지가 확보될 수 있다. 지금 당장 blocking은 아니나, `test_llm_adapter.py`에 추가하면 LLM 어댑터 모듈의 책임 범위 내에서 더 완결된 테스트가 된다.

**2. `MockLLMClient` line 26 미커버**

`responses` 값이 `dict`가 아닌 경우 `{"response": response}`로 감싸는 경로가 테스트되지 않았다. 현재 이 경로를 실제로 사용하는 케이스가 없으므로 제거하거나 테스트를 추가하는 것이 명확하다.

**3. `complete_json`에서 schema system message 중복 가능성**

`complete_json`은 schema 정보를 담은 system 메시지를 무조건 맨 앞에 prepend한다. 호출 측이 이미 system 메시지를 포함하여 넘기면 system 메시지가 두 개가 된다. 현재 MVP 범위에서는 문제가 없으나, 향후 프롬프트가 복잡해질 경우 혼선이 생길 수 있다. 문서화 또는 주석으로 이 동작을 명시해두면 좋다.
