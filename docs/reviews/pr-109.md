# PR #109 Local Review — 인증 refresh 토큰 확장 (이슈 #96, C3/Q8/N2)

## Review Summary

이슈 #96. 기존 access-only 인증에 refresh 토큰 체계를 도입했다. login 응답 확장 + `POST /auth/refresh`
신설 + 만료 env 설정화. 설계 `docs/designs/046-auth-refresh-token.md`와 일치한다. FE
`project_stock_frontend#46`(PR #52)과 페어. 파이프라인: Opus 설계 → Sonnet 구현 → Opus 검증·PR.

검토 항목:

- **토큰 타입 분리(C3)**: access/refresh JWT에 `type` claim(`"access"`/`"refresh"`). `get_current_user`가
  `type != "access"` 거부 → **refresh 토큰으로 보호 API 접근 차단**(`test_get_me_rejects_refresh_token`).
  refresh 엔드포인트는 `type == "refresh"`만 통과. `SECRET_KEY`/`ALGORITHM` 공용 재사용.
- **만료 env(N2)**: `ACCESS_TOKEN_EXPIRE_MINUTES` 30→15, `REFRESH_TOKEN_EXPIRE_MINUTES=2880`(2일). `.env.example`
  반영. `expires_in = ACCESS_TOKEN_EXPIRE_MINUTES * 60`(login/refresh 일관).
- **login 응답 확장(계약 변경)**: `Token`에 `refresh_token`·`expires_in` 추가, 기존 `access_token`/`token_type`
  유지(하위호환). `UserService.login`이 access+refresh 동시 발급.
- **`/auth/refresh`(Q8)**: `RefreshRequest{refresh_token}` → 디코드·`type`·`sub`·사용자 존재 검증 후 새 access.
  만료/변조/access-토큰-제시/미존재 사용자 → 401(`AUTH_INVALID_TOKEN`). 7개 케이스로 단언.
- **비회전 정책(MVP)**: refresh 응답에 새 refresh 미발급(access만 갱신). `refresh_token` 빈 문자열로 노출,
  `test_refresh_returns_new_access_token`이 `assert not data.get("refresh_token")`로 명시. **FE(PR #52)는 빈
  값이면 기존 refresh 유지(`||`)하도록 정합** — 통합 확인 완료.
- **에러 코드**: refresh 실패는 기존 `ErrorCode.AUTH_INVALID_TOKEN` 재사용(신규 코드 미도입, FE는 둘 다
  재로그인이라 구분 불요 — 설계 결정 6 준수).
- **무상태**: JWT라 User 모델·테이블 변경 없음, Alembic 불요(설계 결정 7).

## Blocking

- 없음.

## Suggestions

- **`Token.refresh_token = ""` 기본값(비차단)**: 비회전 응답을 빈 문자열로 노출하는 방식은 동작하나, 의미상
  "미발급"을 빈 문자열로 표현해 약간 모호하다. `str | None = None`(+`exclude_none`)이 더 명시적이나, FE가 빈/
  null 모두 기존 유지로 흡수하므로 현 방식도 안전. 회전 정책 도입 시 재정리 권장.
- **`_INVALID_TOKEN_EXC` 모듈 상수 재사용(확인됨, 비차단)**: `AppException` 인스턴스를 모듈 레벨에서 재사용한다.
  현재 `headers` 등 mutable 상태가 없어 안전하나, 향후 예외에 가변 컨텍스트가 붙으면 인스턴스 공유가 위험할 수
  있다. 팩토리 함수로 두는 편이 방어적.
- **AT 15분 전환 영향(비차단)**: 기존 `ACCESS_TOKEN_EXPIRE_MINUTES` 30→15로 단축. 기존 토큰 의존 테스트/문서가
  30분을 가정하지 않는지 확인됨(`test_config` 갱신). 운영 배포 시 기존 발급 토큰은 만료가 빨라짐(정상).
