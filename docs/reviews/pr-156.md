# PR #156 — 대시보드 추이 시계열 endpoint (feat/be-dashboard-trends)

리뷰어: Claude Code (Opus, 오케스트레이터). 구현: Codex. 브랜치: feat/be-dashboard-trends → main. 관련: BE #155, FE #89, 설계 062.

## Review Summary

설계 062대로 `GET /dashboard/trends?days=N` endpoint를 추가했습니다. 인증 사용자 기준으로 최근 N일 일별 발생 건수를 계열 3종(`risk_alerts`·`review_signals`·`important_news`) 고정 순서로 반환합니다. `DashboardTrendService.get_trends`는 UTC 오늘을 기준으로 윈도우 날짜 목록을 만들고, `start_at`(윈도우 시작 0시)·`end_at`(오늘 다음날 0시) UTC datetime 경계로 범위를 건 뒤 `func.date(created_at)`로 일별 group by 집계합니다. risk/review 계열은 `Alert.created_at`(signals join, signal_type 필터), news 계열은 `AlertCandidate.created_at`(NEWS_SURGE·DISCLOSURE + HIGH)에서 파생하며, `_build_series`가 윈도우 전체 날짜를 결측일 0으로 채웁니다. 라우트는 `days: Query(ge=1, le=90)` 기본 14로 상한 초과 시 422를 반환하고, 기존 summary·briefing과 동일한 인증 컨텍스트를 씁니다. 신규 테이블·마이그레이션은 없습니다.

설계의 핵심 결정인 **mutable-state 필터 제외**가 정확히 반영됐습니다. 세 계열 모두 `Alert.status = UNREAD`·`active_clause`(만료)·`AlertCandidate.status = UNREAD`를 적용하지 않고 `created_at` 발생 건수만 셉니다. 이는 요약 카드가 쓰는 필터와 다르며, 발생 추이가 이후 상태 변화에 흔들리지 않게 하려는 의도입니다.

## Suggestions

**S1 · `func.date()`의 세션 타임존 의존 (비차단)**

일별 버킷팅에 `func.date(created_at)`를 사용합니다. 범위 필터(`created_at >= start_at AND < end_at`)는 UTC datetime 경계라 조회 대상 집합은 정확합니다. 다만 group by 라벨은 DB 세션 타임존을 따르므로, 운영 DB 세션이 UTC가 아니면 경계일(자정 근처) 건이 인접일로 버킷팅될 여지가 있습니다. 현재 테스트 DB·UTC 세션에서는 정확합니다. 후속으로 DB별 UTC 변환(예: Postgres `date_trunc('day', created_at AT TIME ZONE 'UTC')`)으로 엄격히 맞출 수 있습니다. 구현자(Codex)도 동일 리스크를 자체 보고했습니다.
