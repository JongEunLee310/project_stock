# PR #149 로컬 리뷰 — #136 LLMGateway 오케스트레이터 조립 + Phase 1 경계 테스트

## Review Summary

ADR-007이 둔 단일 choke point를 실제 코드로 옮긴 PR입니다. `LLMGateway`가 #134의 라우터와 #135의 프라이버시 게이트를 하나의 진입점으로 묶습니다. 흐름은 task_type으로 provider를 해소하고, 클라우드 경로면 `PrivacyGate.guard`로 페이로드를 검사한 뒤, 검사를 통과한 CloudSafe projection의 `as_payload()` 본문만 transport로 보내는 구조입니다.

설계 `docs/designs/056-llm-gateway.md`와 구현이 정확히 일치합니다. 경계 검사는 transport 선택·호출 전에 게이트웨이 내부에서만 수행되어 우회가 불가능하고(ADR-009 §4), 미구성 provider는 `LLMRoutingError`로 fail-closed 처리됩니다. 클라우드로 나가는 본문은 원본 entity가 아니라 projection의 화이트리스트 필드(구간·불리언)뿐입니다.

#133 타입·#134 라우터·#135 프라이버시와 같은 "도입 후 소비는 후속" 패턴입니다. 살아있는 소비처 이관은 본 PR 범위가 아니며, 게이트웨이는 도입·테스트만 합니다.

## Suggestions

- `get_llm_gateway()`는 `settings.LLM_PROVIDER`가 `cloud`일 때 `local` 슬롯을 `get_llm_client("local")`(`LocalLLMProvider`)로 채웁니다. Phase 1 라우팅 표는 전 작업을 `cloud`로 보내므로 이 슬롯은 아직 도달하지 않지만, 로컬 provider가 실제로 준비되는 시점(`NotImplementedError` 해소)에 이 매핑이 살아있는 경로가 됩니다. 그때 통합 테스트를 함께 두면 좋겠습니다.
- 게이트웨이의 타입 있는 API가 payload를 일괄 `CloudSafePayload`로 받아 로컬 경로에도 원본 entity를 직접 흘릴 수 없습니다. ADR-009가 허용하는 "raw→local"보다 엄격한데, 설계 §5에 이미 의도된 안전 기본값으로 기록되어 있습니다. 로컬 소비처가 생길 때 완화 여부를 재검토하면 됩니다.
