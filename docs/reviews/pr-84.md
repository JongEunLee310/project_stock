## Review Summary

PR #84 — Issue #62 `[BE] 로깅 구조 정리` 구현

핸드오프/설계(042) 범위를 정확히 충족한다. 운영 추적을 위한 기본 로깅 구조를 도입했다. `app/core/logging.py`(신규)가 `setup_logging`으로 `APP_ENV`별 출력(prod=JSON, dev=text)을 구성하고, `contextvar` 기반 request_id를 로그 레코드 팩토리로 모든 레코드에 주입하며, 민감 키 블랙리스트(`authorization`/`api_key`/`secret`/`token`/`password`/`credential`) 기반 `mask_sensitive` + `SensitiveDataFilter`를 제공한다. `app/core/middleware.py`(신규) `RequestIdMiddleware`가 `X-Request-ID`를 수용·생성하고 응답 헤더로 전파하며 요청 완료 로그(method/path/status/latency_ms)를 남긴다. `app/core/exceptions.py`는 AppException(warning)·unhandled(error, 스택 포함)를 request_id와 함께 기록하고 응답 헤더에 request_id를 반영한다. `app/adapters/llm/openai.py`는 provider 호출 시작/타임아웃/실패/완료를 로깅한다(요청 내용은 미기록).

신규 도메인/스키마/엔드포인트 없음, API 응답 envelope 변경 없음. 표준 `logging`만 사용해 외부 로깅 프레임워크 미도입(설계 Decisions 준수). 보호 파일 변경 없음. 로컬 검증 전부 통과(205 passed), CI(`Verify Project Stock`) pass. 블로킹 이슈 없음.

## Suggestions

### 1. 마스킹은 키 기반 — 값에 직접 박힌 시크릿은 마스킹되지 않음 (Minor)

**위치:** `app/core/logging.py:39-49, 99-104`

`mask_sensitive`는 dict의 **키 이름**이 민감 패턴을 포함할 때만 값을 가린다. 따라서 `logger.info("token=%s", raw_secret)`처럼 시크릿을 문자열 위치 인자/메시지에 직접 넣으면 마스킹되지 않는다. 설계 042의 "민감 키 블랙리스트로 시작" 결정과 일치하는 의도된 범위다.

**권장 수정 방향:** 유지하되, "민감 값은 dict/`extra`로 구조화해 넘긴다"는 컨벤션을 #65 통합 문서(로깅 가이드)에 한 줄 명시하면 좋다.

### 2. 예외 핸들러의 request_id 재설정은 BaseHTTPMiddleware contextvar 전파 갭에 대한 적절한 방어 (관찰)

**위치:** `app/core/exceptions.py:113-118`(`_set_request_context_from_state`)

`BaseHTTPMiddleware`는 다운스트림을 별도 컨텍스트에서 실행해 미들웨어가 set한 contextvar가 핸들러까지 일관 전파되지 않을 수 있다. 핸들러가 `request.state.request_id`에서 다시 읽어 로그 동안만 set/reset 하는 방식은 이 갭을 정확히 보완한다 — 적절한 선택.

**권장 수정 방향:** 유지.

### 3. `setup_logging`이 root logger level을 INFO로 고정 (Minor)

**위치:** `app/core/logging.py:75`

root level을 무조건 INFO로 설정한다. 핸들러는 marker로 멱등 교체되어 누수는 없으나, 향후 레벨을 env로 조정하려면 설정값 분리가 필요할 수 있다. 현재 범위에선 문제 없음.

**권장 수정 방향:** 유지. 후속에서 로그 레벨 env 노출이 필요하면 그때 분리.
